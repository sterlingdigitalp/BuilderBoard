# Builder Request Timeline

This document traces a complete Builder request lifecycle, mapping the execution flow from user input to LLM response and UI update, backed by runtime execution traces and system metrics.

## Timeline Flow

### 1. Frontend Event
- **Component:** `src/components/Chat/ChatComposer.tsx`
- **Action:** User inputs a message and presses "Enter" or clicks "Send".
- **React Flow:** The form submission invokes `onSend`, bound to the `sendMessage` callback generated by `usePaneChat.ts`.
- **Latency/Idle:** Near instantaneous (< 5ms). The composer locks state via `isBusy` while the operation is processed.

### 2. IPC Request
- **File:** `src/stores/chatCommands.ts`
- **Action:** The frontend dispatches two IPC commands sequentially:
  1. `message_create`: Synchronously creates the empty `assistant` pending message via `invoke("message_create", { ... })`.
  2. `stream_chat`: Dispatches the async streaming job via `invoke("stream_chat", { ... })`.
- **Latency/Idle:** The IPC serialization/deserialization into Tauri takes under 5ms. The UI remains idle waiting for stream events.

### 3. Backend Routing
- **File:** `src-tauri/src/storage/commands.rs` -> `src-tauri/src/stream_execution.rs`
- **Action:** The Tauri command `stream_chat` invokes `run_background_stream_chat`, which spawns a background runtime task running `run_background_stream_chat_inner`.
- **Duplicated Work:** Minimal. `run_background_stream_chat` merely sets up the thread isolation.

### 4. Planner & Resource Resolution
- **Files:** `src-tauri/src/execution/manager.rs`, `src-tauri/src/stream_execution.rs`
- **Action:**
  - `ExecutionManager::resolve_stream_route` decides the engine and constructs an `ExecutionResolution` determining the profile, effort, and model.
  - Phase **`execution_manager`**: Assigns constraints (e.g. `GPT-5.5`, `openai` engine, effort `high`).
  - Phase **`capability_resolver`**: Evaluates policy rules against available tools. Output indicates 20 tools registered, 20 tools allowed.
- **Latency/Idle:** Context construction and capability scoping execute very quickly (<10ms). No blocking operations.

### 5. Prompt Construction & Filesystem Enrichment
- **File:** `src-tauri/src/stream_execution.rs`
- **Action:** Monitored under phase `stream_chat_prepare`.
  - Resolves database requirements.
  - Pulls context metadata from the `ProjectScopeCache` and optional filesystem states.

### 6. Tool Loop & LLM Request
- **File:** `src-tauri/src/stream_execution.rs` (Loop execution mapping to `OpenAIExecutionEngine`)
- **Action:** The system executes iterations inside a loop where `OpenAIExecutionEngine` interprets LLM chunks mapping to native function names.
- **Traced Phases:**
  - **`loop_round_start`**: Controller starts an interaction (Round 1).
  - **`native_tool_parsing`**: LLM asks to use tools, emitting payloads like `{"name":"filesystem.write","arguments":"{...}"}`.
  - **`loop_round_decision`**: Engine decides to continue (`termination_reason:"tool_calls_present"`).
  - **`tool_registry_lookup`**: Looks up the mapped native Rust function.
  - **`tool_validation`**: Validates the JSON schema. (Blocks if parameters are missing, resulting in `VALIDATION_ERROR`).
  - **`tool_execution_started`**: Triggers the capability.
  - **`tool_permission_check`**: Validates capability (e.g. `write_files`).
  - **`tool_execution_completed`**: Tool writes response, emitting metrics.
- **Latency/Blocking:**
  - Network LLM interaction generates the largest block of time (measured in `ENGINE_REQUEST_DURATION_MS`).
  - Local tool execution measured directly (e.g. `duration_ms: 0` for fast disk writes).
  - The controller loops until a `termination_reason: "no_tool_calls"` is reached.

### 7. Response Processing & Output Streaming
- **File:** `src-tauri/src/stream_execution.rs`, `src-tauri/src/execution/engine.rs`
- **Action:** The chosen Engine resolves stream chunks to `ExecutionEvent::TextDelta` events.
- **Tracing Phase:** `engine_stream`
- **Metric:** Time To First Token (`TTFT_MS`) and total stream duration (`ENGINE_STREAM_TOTAL_MS`).

### 8. UI Update
- **File:** `src/hooks/usePaneChat.ts`, `src/stores/chatCommands.ts`
- **Action:**
  - The frontend captures `message_stream_update` (or chunks directly) via Tauri events.
  - The React hook's `queueStreamDelta` accumulates deltas in an animation frame (`window.requestAnimationFrame`) via `flushPendingDeltas`.
  - Once completion triggers, state displays updated final texts.
- **Duplicated Work:** Combining deltas uses a `Map` structure grouped per frame, minimizing React DOM recalculations.

## Summary

The entire Builder response executes as a pipeline with strict boundaries:
Frontend State -> Tauri IPC -> Planner -> LLM Tool Loop -> Execution Events -> React Render.

Metrics heavily weight time towards Network LLM resolution, while Native Tools execute fast locally without duplication.
